DREAMXEC CODEBASE - ADVANCED SENSITIVE DATA LEAKAGE & PAYLOAD EXPOSURE REPORT
================================================================================

This document outlines all instances where the backend Express API exposes 
sensitive data, leaks private user columns, or sends excessive database fields 
to the React frontend when only a subset is required.

--------------------------------------------------------------------------------
1. CRITICAL VULNERABILITIES: SENSITIVE DATABASE FIELD LEAKS
--------------------------------------------------------------------------------

### 1.1 Profile Update Endpoints Leaking Hashed Passwords & Tokens
*   **File Path:** [server/src/api/profile/profile.controller.js](file:///c:/ongoing_works/codes/Dreamxec/server/src/api/profile/profile.controller.js)
*   **Endpoints:** `PUT /api/profile/me` and `POST /api/profile/picture`
*   **Exposed Fields:** "password" (hashed), "verificationToken", "verificationTokenExpiry", "resetPasswordToken", "resetPasswordExpiry".
*   **The Issue:** The backend performs updates using Prisma client commands and returns the raw updated records directly to the client payload:
    - Donor updates (Lines 234-258)
    - User updates (Lines 333-348)
    - Profile picture updates (Lines 369-388)
*   **Potential Risk:** Any inspection of network requests (or malicious third-party script running in the browser) can harvest the hashed password (which can be cracked offline) and the active reset password/verification tokens (which can be used to hijack user accounts).
*   **How to Fix:** Project only safe columns via Prisma's `select` block or manually clean the updated object before response serialization:
    ```javascript
    const cleanProfile = { ...updated };
    delete cleanProfile.password;
    delete cleanProfile.verificationToken;
    delete cleanProfile.verificationTokenExpiry;
    delete cleanProfile.resetPasswordToken;
    delete cleanProfile.resetPasswordExpiry;
    ```

### 1.2 Authentication Helper exposing Active System Tokens
*   **File Path:** [server/src/api/auth/auth.controller.js](file:///c:/ongoing_works/codes/Dreamxec/server/src/api/auth/auth.controller.js)
*   **Endpoint:** `POST /api/auth/login`
*   **Exposed Fields:** "verificationToken", "resetPasswordToken", and their expiry fields.
*   **The Issue:** The `createAndSendToken` helper (Lines 29-41) sets the hashed password to `undefined` but fails to strip other system-level security tokens from the returning user profile:
    ```javascript
    user.password = undefined;
    ```
*   **Potential Risk:** Active tokens are sent to the client browser on every login, increasing exposure to token harvesting.
*   **How to Fix:** Set all token fields to `undefined` inside the helper before calling `res.json`.

2. MASSIVE PAYLOAD BLOAT & STRUCTURAL OVER-EXPOSURE (LIST VIEWS)
--------------------------------------------------------------------------------

### 2.1 Public Campaigns List exposing Full Descriptions, FAQs, & Team Members
*   **File Path:** [server/src/api/user-projects/user-project.controller.js](file:///c:/ongoing_works/codes/Dreamxec/server/src/api/user-projects/user-project.controller.js)
*   **Endpoint:** `GET /api/user-projects/public` (Lines 467-518)
*   **The Issue:** The Prisma query `prisma.userProject.findMany` lacks a root-level `select` block. It retrieves and sends the complete database document for every project.
*   **Frontend Usage Mismatch:** The React frontend campaign card (**[CampaignCard.tsx](file:///c:/ongoing_works/codes/Dreamxec/client/src/components/CampaignCard.tsx)**) only displays:
    - Campaign `title` and `imageUrl`
    - College name and Club name (`club.college`, `club.name`)
    - Goals and progress (`goalAmount`, `amountRaised`/`currentAmount`)
    - Star `rating`
    - Count of milestones (`milestones.length`)
*   **Unnecessary Data Sent:** The backend sends the full `description` (multi-kilobyte markdown text), the complete `teamMembers` list, `faqs` list, `presentationDeckUrl`, and `campaignMedia` arrays for every project in the browse list.
*   **Potential Risk:** Extreme network payload bloat when browsing projects, leading to slower page loads and higher data usage for users, especially on mobile connections.
*   **How to Fix:** Restrict the query to only fetch card-relevant metadata:
    ```javascript
    const rawProjects = await prisma.userProject.findMany({
      where: { status: 'APPROVED' },
      select: {
        id: true,
        title: true,
        imageUrl: true,
        goalAmount: true,
        amountRaised: true,
        rating: true,
        userId: true,
        createdAt: true,
        club: { select: { id: true, name: true, college: true } },
        milestones: { select: { id: true } } // Fetch only IDs to compute count
      }
    });
    ```

3. USER PRIVACY & FUNCTIONAL LEAKAGE IN OPPORTUNITIES APPLICANTS COUNT
--------------------------------------------------------------------------------

### 3.1 Exposing Application Lists vs. Broken Applicants Counter
*   **File Path:** [server/src/api/donor-projects/donor-project.controller.js](file:///c:/ongoing_works/codes/Dreamxec/server/src/api/donor-projects/donor-project.controller.js)
*   **Endpoint:** `GET /api/donor-projects/public` (Lines 218-232)
*   **The Issue:** The frontend opportunity card (**[BrowseProjects.tsx](file:///c:/ongoing_works/codes/Dreamxec/client/src/components/BrowseProjects.tsx#L164)**) attempts to display the total count of applications via `project.interestedUsers?.length`. However, the backend does not join or count applications in `getPublicDonorProjects`. The client mappers init it to `interestedUsers: []`, making the card constantly display `0 students applied`.
*   **The Risk:** If a developer attempts to fix this by blindly joining `applications: true` in the find query, it will leak all students' private application details (like cover letters, emails, names) publicly to anyone browsing the cards.
*   **How to Fix:** Fetch the relation count securely and performantly without loading database records using Prisma's `_count` aggregation:
    ```javascript
    const donorProjects = await prisma.donorProject.findMany({
      where: { status: 'APPROVED' },
      include: {
        donor: { select: { id: true, name: true, organizationName: true } },
        _count: {
          select: { applications: true } // Fetches only the count of applicants safely!
        }
      }
    });
    ```
    Then map `_count.applications` to the client payload as a safe count integer.

4. SECURE PROTOCOLS & NETWORK LEAKS
--------------------------------------------------------------------------------

### 4.1 JWT Authentication Tokens exposed in Browser Query Parameters
*   **File Path:** [server/src/api/auth/auth.controller.js](file:///c:/ongoing_works/codes/Dreamxec/server/src/api/auth/auth.controller.js)
*   **Endpoints:** `GET /api/auth/google/callback` and `GET /api/auth/linkedin/callback` (Lines 272-350)
*   **The Issue:** After successful third-party login (Google/LinkedIn), the backend redirects users to the frontend with the active authentication token in the URL:
    ```javascript
    res.redirect(`${process.env.CLIENT_URL}/auth/callback?token=${token}`);
    ```
*   **Potential Risk:** Active tokens in query strings are saved in browser history, visible to shoulder-surfing, and leaked to third-party CDNs or scripts via HTTP `Referer` headers.
*   **How to Fix:** Use secure HTTP-only cookies to pass authorization sessions, or use window messaging (`postMessage`) from a pop-up window to transfer the token securely inside the memory space.

8. STUDENT VERIFICATION KYC BYPASS AND REVOCATION FAILURE
--------------------------------------------------------------------------------

### 8.1 Critical KYC Check Bypass on Post-Payment Document Upload
*   **File Path:** [server/src/api/studentVerification/studentverfication.controller.js](file:///c:/ongoing_works/codes/Dreamxec/server/src/api/studentVerification/studentverfication.controller.js)
*   **Endpoint:** `POST /api/student-verification/verify` (Line 52)
*   **The Issue:** 
    - The transaction automatically updates user flags to `studentVerified: true` and `canCreateCampaign: true` BEFORE the admin reviews the uploaded identification files.
    - The verification request status is created as `VERIFIED` instead of `PENDING` (Line 117).
    - In `rejectStudentVerification`, rejecting a request sets request status to `REJECTED` but fails to revoke user flags `studentVerified` and `canCreateCampaign`.
*   **How to Fix:** Keep user flags `false` and request status `PENDING` upon verify callback, and implement user flag updates (revoking privileges to false) upon verification rejection.
